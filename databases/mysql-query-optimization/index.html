MySQL Query Optimization

Best Practices

  • Always have a primary key. If you omit a primary key, MySQL will automatically generate one.
  • Create single indexes with multiple columns; order columns in indexes from left-to-right. The Primary Key is automatically added to custom indexes, so do not include it manually. Every index you create duplicates the primary key, so limit them.
  • Index equality before ranges. When writing queries, try to use equality constraints first.
  • Avoid duplicate indexes in tables and queries: pt-duplicate-key-checker
  • Do not mix data types and do not rely on automatic type conversion. Type conversions will hobble the Optimizer.
  • Negative clauses / subqueries are slower than positive clauses, eg. IS NOT, NOT IN, NOT LIKE.
  • Prefer INNER JOIN to LEFT JOIN. LEFT, RIGHT, and FULL joins require scanning the entirety of their respective tables.
  • UNION ALL is preferable to multiple UNIONs for the sake of the Optimizer.
  • ORDER BY can be slow; if you're ordering a very small amount of data, do it in the application.
  • ORDER BY and LIMIT will pull all of the data into memory; try to paginate in the application.
  • ProxySQL allows you to rewrite queries when they reach the database.

Finding Slow Queries

  • Slow Query Log; n <= 500ms is the threshold for an unacceptably slow query.
  • PMM/QAN
  • Network sniff
  • 3rd Party Tooling; Vividcortext / Solarwinds / etc.

Debugging Slow Queries

  • MySQL supports the EXPLAIN statement which will reveal the steps taken to resolve a query.
  • The Performance Schema tracks events in the MySQL engine and can provide various debug data.
  • Running an Optimizer Trace can tell you if the engine is using an Index Scan or a Full-Table Scan for a query.

PMM/QAN

Peracona Monitoring and Management / Query Analytics

PMM Qan is an analytics app which will reveal information like the load and latency on a system as a query is run, how many times any particular query is run, the fingerprint, and the EXPLAIN.

Slow Index Lookups

Indexes are maintained as a double-linked lists, binary tree data structure.

  • Low cardinality / large data set; The point of working with indexes is to reduce the number of items to scan through.
  • multiple index traversal; Scanning through more than one index can be slow.
  • index column used as argument
  • looking for suffix; Looking for a prefix is more common.
  • non-leading column lookup
  • data type mismatch; Do not mix data types. Do not store integers as their string counterparts.
  • Character Set / Collation mismatch
  • MySQL bug

The Optimizer

The Optimizer is a MySQL daemon which statistically analyzes queries to optimize how joins should be executed.

You can add index hints, globally or in query blocks, to help the optimizer guess which indexes to use.
